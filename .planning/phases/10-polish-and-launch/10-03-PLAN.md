  <name>Task 1: Vercel deployment config</name>
  <files>vercel.json</files>
  <read_first>package.json (scripts section)</read_first>
  <action>
Create vercel.json at the project root with:

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "buildCommand": "pnpm build",
  "framework": "nextjs",
  "headers": [
    {
      "source": "/sw.js",
      "headers": [
        { "key": "Cache-Control", "value": "no-cache, no-store, must-revalidate" },
        { "key": "Service-Worker-Allowed", "value": "/" }
      ]
    },
    {
      "source": "/manifest.webmanifest",
      "headers": [
        { "key": "Content-Type", "value": "application/manifest+json" }
      ]
    }
  ]
}
```

Key points:
- `buildCommand: "pnpm build"` ensures the `--webpack` flag from package.json scripts is used (Serwist requires webpack, not Turbopack)
- Service worker headers ensure no caching of sw.js and correct scope
- Manifest gets correct content type
- Do NOT include env vars in vercel.json — those are set in Vercel dashboard. The required env vars are documented in CONTEXT.md: TURSO_DATABASE_URL, TURSO_AUTH_TOKEN, CRON_SECRET, BETTER_AUTH_SECRET, BETTER_AUTH_URL, VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, NEXT_PUBLIC_VAPID_PUBLIC_KEY, NEXT_PUBLIC_BASE_URL
- No NEXT_PUBLIC_ secrets — NEXT_PUBLIC_VAPID_PUBLIC_KEY is a public key (safe), NEXT_PUBLIC_BASE_URL is just the domain
  </action>
