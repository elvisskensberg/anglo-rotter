<task type="auto">
  <name>Task 1: Install Better Auth and configure server + client + schema</name>
  <files>package.json, src/lib/auth.ts, src/lib/auth-client.ts, src/db/schema.ts</files>
  <action>
1. Install better-auth: `pnpm add better-auth`

2. Create `src/lib/auth.ts` — Better Auth server configuration:
   - Import `betterAuth` from "better-auth"
   - Import the Drizzle adapter from "better-auth/adapters/drizzle" (check Better Auth docs — the adapter import path may be "better-auth/adapters/drizzle" or configured inline)
   - Import `db` from "@/db/client"
   - Configure with:
     - `database: drizzleAdapter(db)` (or the equivalent Drizzle integration pattern)
     - `emailAndPassword: { enabled: true }` for email/password auth (per D-01 CONTEXT.md)
     - `session: { expiresIn: 60 * 60 * 24 * 7 }` (7 day sessions for AUTH-02 persistence)
   - Export `auth` and also export `GET` and `POST` handlers for the route: `export const { GET, POST } = auth.handler`
   - IMPORTANT: Check `npm show better-auth version` first to confirm the API. Better Auth's API may use `betterAuth()` or `createAuth()`. Consult the package's README/types for the exact function name and adapter pattern.

3. Create `src/lib/auth-client.ts` — Better Auth React client:
   - Import `createAuthClient` from "better-auth/react"
   - Create client: `export const authClient = createAuthClient({ baseURL: process.env.NEXT_PUBLIC_BETTER_AUTH_URL || "" })`
   - Export destructured: `export const { useSession, signIn, signUp, signOut } = authClient`

4. Update `src/db/schema.ts` — Add Better Auth required tables:
   - Better Auth with Drizzle adapter requires specific table names. Check the Better Auth Drizzle adapter docs for the exact schema.
   - Typically needs: `user` (may conflict with existing `users` table), `session`, `account`, `verification`
   - The existing `users` table has fields (username, memberSince, postCount, starRating, ratersCount, points). Better Auth needs (id, name, email, emailVerified, image, createdAt, updatedAt).
   - Strategy: Rename existing `users` to `user` if Better Auth requires it, OR keep `users` as a profile table and let Better Auth create its own `user` table. Preferred approach: Let Better Auth own `user` table with its required fields, then ADD the forum-specific columns (username text unique, postCount integer default 0, starRating integer default 1, ratersCount integer default 0, points integer default 0) directly onto that table. This way Better Auth's auth works AND forum profile data lives on the same row.
   - Add `session` table: id (text pk), expiresAt (integer), token (text unique), ipAddress (text), userAgent (text), userId (text references user.id)
   - Add `account` table: id (text pk), accountId (text), providerId (text), userId (text references user.id), accessToken (text), refreshToken (text), idToken (text), accessTokenExpiresAt (integer), refreshTokenExpiresAt (integer), scope (text), password (text), createdAt (integer), updatedAt (integer)
   - Add `verification` table: id (text pk), identifier (text), value (text), expiresAt (integer), createdAt (integer), updatedAt (integer)
   - CRITICAL: Verify exact column names and types against Better Auth's Drizzle schema generation. Run `npx better-auth generate --output ./drizzle` or check docs for the canonical schema. The column names must match exactly or Better Auth will fail silently.
   - Update all existing code that references `users` table — the thread API route imports `users` from schema. If renaming, update those imports.
   - Keep the `memberSince` field as a computed value from `createdAt` instead of a separate column.

5. Add env vars to `.env` (or `.env.local`):
   - `BETTER_AUTH_SECRET` — generate a random 32+ character string
   - `BETTER_AUTH_URL=http://localhost:3000` (for dev)
   - `NEXT_PUBLIC_BETTER_AUTH_URL=http://localhost:3000`

IMPORTANT: After installation, run `pnpm build` to verify no type errors. If Better Auth's API differs from what's described above, adapt to the actual API — the package version may have evolved.

<task type="auto">
  <name>Task 2: Auth API route and middleware</name>
  <files>src/app/api/auth/[...all]/route.ts, src/middleware.ts</files>
  <action>
1. Create `src/app/api/auth/[...all]/route.ts`:
   - Import auth handlers from `@/lib/auth`
   - Export `{ GET, POST }` from the auth handler
   - This is the standard Better Auth catch-all pattern — all auth requests (/api/auth/sign-in, /api/auth/sign-up, /api/auth/sign-out, /api/auth/session) are handled by this single route

2. Create `src/middleware.ts` (Next.js middleware at project root src/):
   - Import `NextResponse` from "next/server" and `NextRequest`
   - Import `auth` from "@/lib/auth" (or use Better Auth's middleware helper if available)
   - Define `config.matcher` to protect write routes only:
     - `/api/threads` (POST only — creating threads)
     - `/api/threads/*/replies` (POST only — creating replies)
     - `/api/threads/*/rate` (POST only — rating threads)
   - The middleware should check for a valid session cookie. If no session, return 401 JSON response for API routes.
   - Public routes (all GET requests, /api/auth/*, /api/forums/*, /api/news, etc.) must NOT be blocked.
   - IMPORTANT: Better Auth may provide its own middleware helper. Check the docs. If it provides `authMiddleware` or similar, use that. Otherwise, manually check the session cookie.
   - Matcher config: `export const config = { matcher: ["/api/threads/:path*"] }` — then inside the middleware function, only block POST/PUT/DELETE methods while allowing GET through.

3. Verify the auth flow works end-to-end by confirming the API route responds:
   - `curl http://localhost:3000/api/auth/session` should return a session object (empty/null if not logged in, but should NOT 404 or 500)
